    33→### 🟠 HIGH — ExerciseVideoPlayer.tsx:164-167 — Memory leak of object URL on rapid URL change
    34→
    35→**Category:** leak
    36→
    37→**File:** `src/components/exercise/ExerciseVideoPlayer.tsx` lines 151-173
    38→
    39→**Description:**
    40→The Cache API effect creates an object URL via `URL.createObjectURL(blob)` inside an async IIFE. If the user navigates away (or the `activeUrl` prop changes) while the `responseToObjectUrl` promise is in flight, the cleanup sets `cancelled = true`. The async function then returns early AFTER the URL has already been created:
    41→
    42→```typescript
    43→const objUrl = await responseToObjectUrl(response);
    44→if (cancelled) return;   // ← objUrl is leaked: never set to state, never revoked
    45→if (objUrl) setBlobUrl(objUrl);
    46→```
    47→
    48→When `cancelled` is true on this branch, `objUrl` was created (holding a reference to a Blob in memory) but is never stored in React state, so the separate `useEffect` cleanup at line 176-180 (which only revokes `blobUrl` from state) never sees it. The blob URL leaks until the document is unloaded.
    49→
    50→In a long workout-session page with many compact cards (each hosting its own player), rapid scrolling can trigger many such leaks.
    51→
    52→**Suggested fix:**
    53→```typescript
    54→const objUrl = await responseToObjectUrl(response);
    55→if (cancelled) {
    56→  if (objUrl) URL.revokeObjectURL(objUrl);  // ← revoke orphaned URL
    57→  return;
    58→}
    59→if (objUrl) setBlobUrl(objUrl);
    60→setIsLoadingFromCache(false);
    61→```

   168→### 🟡 MEDIUM — CalendarPage.tsx:116-132 — `handleDayClick` has a race condition + re-queries the whole month
   169→
   170→**Category:** bug / race
   171→
   172→**File:** `src/pages/CalendarPage.tsx` lines 116-132
   173→
   174→**Description:**
   175→```typescript
   176→const handleDayClick = useCallback(async (dateKey: string) => {
   177→  setSelectedDateKey(dateKey);
   178→  setSelectedDaySessions([]);
   179→  const monthSessions = await getSessionsByMonth(year, month);
   180→  const daySessions = monthSessions.filter((s) => { ... });
   181→  setSelectedDaySessions(daySessions);
   182→}, [year, month]);
   183→```
   184→
   185→Two issues:
   186→
   187→1. **Race:** If the user taps day A, then day B before A's `getSessionsByMonth` resolves, A's result arrives second and overwrites `selectedDaySessions` — but `selectedDateKey` was already updated to B by the second tap. The drawer for B briefly shows A's sessions.
   188→
   189→2. **Wasted work:** Each tap re-runs a full-table Dexie scan via `getCompletedSessions().filter(...)` even though `getMonthActivitySummary` already did the same scan moments earlier. The result isn't cached anywhere.
   190→
   191→**Suggested fix:** Track the latest clicked dateKey in a ref and ignore stale results:
   192→
   193→```typescript
   194→const lastClickRef = useRef<string | null>(null);
   195→
   196→const handleDayClick = useCallback(async (dateKey: string) => {
   197→  lastClickRef.current = dateKey;
   198→  setSelectedDateKey(dateKey);
   199→  setSelectedDaySessions([]);
   200→  const monthSessions = await getSessionsByMonth(year, month);
   201→  if (lastClickRef.current !== dateKey) return;  // stale
   202→  const daySessions = monthSessions.filter((s) =>
   203→    localDateKey(new Date(s.date)) === dateKey
   204→  );
   205→  setSelectedDaySessions(daySessions);
   206→}, [year, month]);
   207→```
   208→
   209→(Could also cache monthSessions at the page level to avoid the rescan.)

   272→### 🟢 LOW — analytics.ts:21-26 — `getWeekKey` uses a non-standard week-number heuristic
   273→
   274→**Category:** data
   275→
   276→**File:** `src/db/analytics.ts` lines 21-26
   277→
   278→**Description:**
   279→```typescript
   280→const weekNum = Math.ceil((days + startOfYear.getDay() + 1) / 7);
   281→```
   282→
   283→This is a simplified formula, not ISO 8601. It can produce week numbers > 52 (or even > 53 in some years) and doesn't align week 1 with the first Thursday. Because the key is `${year}-W${weekNum}`, the same week can theoretically get two different keys depending on which date you query from (e.g., a Sunday vs. a Tuesday might land in different computed weeks near year boundaries). Sessions in early January could be bucketed into the previous year's last week or vice versa.
   284→
   285→This appears to be pre-existing (not introduced by A1), but since A1 added the `countsInVolume` filter on the same code path, it's worth flagging.
   286→
   287→**Suggested fix:** Use a small ISO-week helper:
   288→```typescript
   289→function getWeekKey(date: Date): string {
   290→  const d = new Date(Date.UTC(date.getFullYear(), date.getMonth(), date.getDate()));
   291→  const dayNum = (d.getUTCDay() + 6) % 7;
   292→  d.setUTCDate(d.getUTCDate() - dayNum + 3);
   293→  const firstThursday = new Date(Date.UTC(d.getUTCFullYear(), 0, 4));
   294→  const weekNum = 1 + Math.round(((d.getTime() - firstThursday.getTime()) / 86400000 - 3 + ((firstThursday.getUTCDay() + 6) % 7)) / 7);
   295→  return `${d.getUTCFullYear()}-W${String(weekNum).padStart(2, "0")}`;
   296→}
   297→```

   301→### 🟢 LOW — recoveryTracker.ts:389-394 — `hoursLeft` can be negative or NaN with future-dated / malformed sessions
   302→
   303→**Category:** data / edge-case
   304→
   305→**File:** `src/services/recoveryTracker.ts` lines 378-398
   306→
   307→**Description:**
   308→```typescript
   309→const hoursSince = (now - datum.lastTrained) / (1000 * 60 * 60);
   310→const recoveryPercent = Math.min(100, Math.round((hoursSince / recoveryHours) * 100));
   311→// ...
   312→const hoursLeft = Math.ceil(recoveryHours - hoursSince);
   313→label = `Just trained · ${hoursLeft}h left`;
   314→```
   315→
   316→If a session has a future `date` (clock skew, manual edit, sync from another device), `hoursSince` is negative, `recoveryPercent` is negative, and `hoursLeft = ceil(recoveryHours - negative) = large positive`. The label would say "Just trained · 96h left" for a session dated 2 days in the future.
   317→
   318→If `session.date` is malformed and `new Date(session.date).getTime()` returns `NaN`, then `hoursSince = NaN`, `recoveryPercent = NaN`. The `if (recoveryPercent < 33)` check is false for NaN, `< 67` is also false, so status falls through to `"recovered"` with `recoveryPercent: NaN`. Bizarre but doesn't crash.
   319→
   320→**Suggested fix:**
   321→```typescript
   322→const hoursSince = (now - datum.lastTrained) / (1000 * 60 * 60);
   323→if (!isFinite(hoursSince) || hoursSince < 0) {
   324→  // Treat future-dated / malformed sessions as "just trained"
   325→  result.set(muscleId, { ...justTrainedEntry });
   326→  continue;
   327→}
   328→```

   332→### 🟢 LOW — DaySessionsDrawer.tsx:84-90 — `dateKey.split("-").map(Number)` doesn't validate shape
   333→
   334→**Category:** error-handling
   335→
   336→**File:** `src/components/calendar/DaySessionsDrawer.tsx` lines 82-91
   337→
   338→**Description:**
   339→```typescript
   340→const [y, m, d] = dateKey.split("-").map(Number);
   341→return new Date(y, m - 1, d).toLocaleDateString(...);
   342→```
   343→
   344→If `dateKey` is malformed (e.g., empty string, missing parts, non-numeric), `y/m/d` will be `undefined` or `NaN`, and `new Date(undefined, NaN, undefined)` produces an Invalid Date. `toLocaleDateString(...)` then returns the string `"Invalid Date"`, which is rendered in the drawer header. Not a crash, but unprofessional UX.
   345→
   346→In practice, `dateKey` always comes from `formatDateKey` in `CalendarGrid.tsx`, which always produces a well-formed `YYYY-MM-DD` string, so this is purely defensive.
   347→
   348→**Suggested fix:**
   349→```typescript
   350→const parts = dateKey.split("-").map(Number);
   351→if (parts.length !== 3 || parts.some((n) => !Number.isFinite(n))) {
   352→  return dateKey;
   353→}
   354→const [y, m, d] = parts;
   355→return new Date(y, m - 1, d).toLocaleDateString(...);
   356→```
